## 1. JSP(Java Server Pages)의 정의 및 특징 🔍
* **개념**: 서블릿 기술과 목적은 같으나, 브라우저에 보내는 HTML 데이터를 만들어내는 데 더 특화된 기술입니다.
* **동작 원리**: JSP는 내부적으로 서블릿과 같은 방식의 코드로 변환되어 실행됩니다.
* **서블릿과의 차이**:
    * **서블릿**: 자바 코드 안에 HTML 코드를 추가하는 방식입니다.
    * **JSP**: HTML 내에 자바 코드를 추가하는 방식이며, 서블릿 컨테이너가 객체를 생성하고 생명 주기를 관리합니다.

## 2. 데이터 전송 방식: GET vs POST 📊
사용자가 입력한 데이터를 서버로 전달하는 방식은 크게 두 가지로 나뉩니다.

| 구분 | GET 방식 | POST 방식 |
| :--- | :--- | :--- |
| **주 용도** | 원하는 데이터 조회, 입력 화면 호출 | 등록, 수정, 삭제와 같은 처리 작업 |
| **데이터 전달** | URL 뒤에 **쿼리 스트링**(`?key=value`)으로 전달 | HTTP 메시지 **Body**에 데이터를 담아 전달 |
| **특징** | 주소창에 데이터가 노출됨 | 주소창에 데이터가 보이지 않음 |
| **한계** | 브라우저에 따라 길이 제한이 있음 | 주소창만으로는 테스트가 어려움 |



## 3. EL(Expression Language) 활용 ⭐
서버에서 데이터를 출력할 때 사용하는 `${ }` 문법의 핵심 내용입니다.

* **파라미터 추출**: `${param.name}` 형식을 사용하여 요청에 전달된 파라미터를 쉽게 추출할 수 있습니다.
* **자동 형변환 (Auto Casting)**:
    * 웹의 파라미터는 모두 문자열로 전달되지만, EL은 산술 연산 시 자동으로 숫자로 변환합니다.
    * 예시: `${param.num1 + param.num2}` 처럼 작성해도 EL이 내부적으로 숫자로 변환하여 계산을 수행합니다.
* **비교**: 책의 예제처럼 `Integer.parseInt()`를 사용하는 것은 JSP 내부에 복잡한 코드를 넣는 것이 적합하지 않음을 보여주는 사례입니다.

## 4. JSP의 한계와 MVC 패턴의 등장 🚩
JSP에서 직접 파라미터를 처리하고 로직을 수행할 때 발생하는 문제를 해결하기 위한 방향성입니다.

* **발생 가능 문제**:
    1. **500 에러**: 파라미터 없이 결과 페이지(`calcResult.jsp`)에 직접 접근하면 `parseInt()` 등에서 예외가 발생합니다.
    2. **유지보수 저하**: 유지보수 측면에서 문제가 발생하므로 최근에는 JSP를 제한적인 용도로 사용합니다.
* **해결책 (웹 MVC 방식)**:
    * **JSP**: 화면을 구성하거나 처리 결과를 보여주는 용도로만 사용합니다.
    * **서블릿**: 쿼리 스트링이나 파라미터 처리를 서블릿을 통해 수행합니다.
    * **구조**: 브라우저는 직접 JSP를 호출하지 않고 서블릿 경로를 통해서 JSP를 보는 방식을 권장합니다.
